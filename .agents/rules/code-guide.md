醫療知識庫與考古題練習系統 (系統設計文件 SDD)1. 系統概述與願景本系統旨在建構一個共筆知識庫與考古題刷題雙向聯動生態系。系統需具備將非結構化文檔（PDF、Word、純圖片）轉化為結構化 JSON 格式的資料萃取管線，並透過多維度標籤與關聯式資料庫，實現知識點與考題的精準檢索。核心功能包含動態權重複習演算法、檢索增強生成（RAG）知識搜尋、多階層權限審核機制，以及支援 Markdown、PDF 與 Notion 的資料匯出引擎。2. 系統架構與技術選型前端架構採用 Next.js 搭配 Tailwind CSS 進行介面開發，全域狀態管理使用 Zustand，並透過 Framer Motion 處理平滑過場動畫。後端核心建立於 Node.js 與 TypeScript 環境，API 傳輸層建議導入 tRPC 以實現前後端型別安全，或使用標準 RESTful API 搭配 Zod 進行負載驗證。主資料庫採用 PostgreSQL 並原生啟用 pgvector 擴充套件以支援向量儲存與餘弦相似度計算。靜態資源與圖片檔案嚴禁存入關聯式資料庫，統一寫入自建之 MinIO 物件儲存伺服器，並於外層掛載 Cloudflare CDN 節點消化讀取流量。AI 解析核心指定呼叫 Gemini 1.5 Pro 多模態 API。3. 資料庫與實體關聯模型 (Schema Design)資料庫設計需遵循嚴格的正規化與外鍵關聯，核心實體分為以下區塊：Users 與權限系統： 導入基於角色的存取控制（RBAC）。建立 Users 表（包含帳號、建立時間）與 Roles、Permissions 表。權限層級分為一般使用者（具備刷題、匯出、提出編輯請求權限）、版主審核員（具備核准編輯請求、修改內容權限）與系統管理員。Wiki_Articles (共筆知識庫)： 儲存結構化條目。欄位包含 UUID、標題、純 Markdown 格式的內容字串、建立者外鍵與發布狀態。圖片一律以 Markdown 語法封裝 CDN 網址存入內文。Questions (考古題庫)： 欄位包含 UUID、年份、考卷類型、支援 Markdown 的題幹字串、使用 JSONB 格式儲存的選項鍵值對、正確解答字串、支援 Markdown 的詳解字串，以及綁定至 Wiki_Articles 的外鍵以實現雙向聯動。另需包含 image_urls 字串陣列欄位供前端預載圖片。Tags (多維度標籤系統)： 建立獨立的 Tags 表，透過關聯表綁定至題目或條目。標籤強制劃分為 4 個維度：基礎學科（如解剖學）、臨床系統（如心血管系統）、題目來源（如特定年份考卷）、題型與狀態（如常考題、圖片題）。User_Question_Records (刷題紀錄)： 記錄 UUID、使用者外鍵、題目外鍵、使用者作答字串、是否答對之布林值與作答時間戳記，作為計算間隔重複演算法的基礎資料。審核與擴充表： 建立 Comments 表處理條目與題目的留言互動；建立 Edit_Requests 表紀錄使用者提出的修改提案與審核狀態；建立 User_Integrations 表安全儲存 Notion API 等第三方存取權杖。4. 核心演算法與 RAG 檢索機制系統出題引擎採用混合式動態權重演算法。透過 PostgreSQL 自訂函數或查詢運算式，在撈取題目的當下即時計算優先級分數。演算法需綜合 3 項變數：第一項為借鑒 FSRS 演算法的指數衰減函數，依據 User_Question_Records 中的連續答對次數與距離上次作答的時間差計算基礎權重；第二項為動態標籤加權，依據前端傳入的考試範圍標籤與常考題 Meta 標籤給予常數加成；第三項為社群難度校準，運用威爾遜區間估計（Wilson score interval）計算全體作答正確率，提升高鑑別度錯題的推播機率。針對 RAG 檢索增強生成，後端需實作背景非同步任務。當 Wiki 條目或 Question 發布時，系統呼叫 text-embedding-004 等嵌入模型，將 Markdown 文本轉化為浮點數陣列，並寫入 PostgreSQL 的 pgvector 欄位。檢索時，將使用者查詢字串向量化後，透過 SQL 運算餘弦相似度找出語意最接近的知識段落，解決傳統 LIKE 語法無法理解醫學同義詞的限制。5. 資料萃取管線與審核工作站 (Data Extraction Pipeline)為確保資料 100% 正確性，文檔匯入需遵循圖文分離、AI 辨識與人工校對 3 階段管線：檔案分流與圖文分離： 若上傳檔案為 Word，透過 mammoth.js 攔截圖片緩衝區上傳至 MinIO，並將內文轉為帶有圖片網址的 Markdown。若為 PDF 或純圖片，嚴格禁止依賴傳統文字提取，必須透過 pdf2pic 或 pdf-poppler 等工具將 PDF 每一頁強制轉譯為 300dpi 的 PNG 高解析度圖片。多模態 AI 結構化萃取： 將分離後的圖片或文字負載傳送至 Gemini 1.5 Pro。系統提示詞需嚴格約束輸出格式為包含題幹、選項與解答的 JSON 陣列。並指示 AI 具備空間感知能力以對抗雙欄排版，同時要求 AI 若偵測到題目包含影像，必須在題幹 JSON 中插入 [需要手動截圖_編號] 的精確佔位符，嚴禁 AI 自行解讀圖片細節。人工校對與裁切站 (Staging Area)： 導入 Human-in-the-Loop 機制。前端實作左右雙欄介面，左側渲染支援雙指縮放的原檔圖片，右側渲染 AI 生成之草稿表單。整合 React-Image-Crop 套件，當審核員在左側框選圖片區域時，觸發前端 Canvas API 截取二進位資料並背景上傳至 MinIO，取得網址後自動替換右側表單內的佔位符，確認無誤後始可寫入 PostgreSQL。6. API 介面設計規範前後端溝通嚴格採用 JSON 格式，所有寫入操作需經過 Zod 驗證層。核心端點包含：GET /api/questions：接收標籤或年份查詢參數，回傳符合條件之題目清單。POST /api/questions/submit：接收題目 ID 與作答狀態，觸發資料庫權重更新與紀錄寫入。GET /api/wiki/:id：回傳特定條目之 Markdown 內文，並運用關聯模型夾帶對應的歷史考古題陣列。POST /api/parser/upload：接收表單檔案上傳，回傳非同步任務 ID（Job ID）。GET /api/parser/status/:job_id：提供前端輪詢機制，追蹤 AI 解析與圖文分離之處理進度，避免大檔案造成 HTTP 請求超時。7. 前端交互邏輯與介面定義儀表板 (Dashboard)： 包含視覺化熱力圖呈現過去 30 天刷題活躍度。提供待複習錯題佇列狀態，以及可複選多維度標籤的動態測驗卷生成模組。測驗引擎 (Quiz Engine)： 採用單題卡片式無干擾介面。整合 Mousetrap 綁定全域快捷鍵操作（數字鍵選答、空白鍵確認）。答錯時透過 Framer Motion 展開下方詳解區塊，並提供跳轉至對應 Wiki 條目的快捷按鈕。雙向聯動閱讀介面： 採用可拖曳雙欄佈局。左側渲染高資訊密度 Markdown，右側邊欄透過 Intersection Observer 監聽左側閱讀深度，動態載入並渲染與當前視窗段落相關聯的考古題。實作浮動工具列，允許反白文字觸發向量搜尋或建立筆記。8. 匯出與外部整合系統實作 3 個層次的資料分發機制。第一層為 Markdown 原始碼匯出，後端將查詢到的紀錄拼接為純文字檔回傳。第二層為客製化 PDF 匯出，透過 Puppeteer 無頭瀏覽器在伺服器端將 Markdown 渲染為 HTML 後列印為附帶 QR Code 連結的實體測驗卷。第三層為 Notion API 整合，透過 @notionhq/client 將系統的多維度標籤與內容區塊，映射並寫入使用者授權的 Notion Database 中。
